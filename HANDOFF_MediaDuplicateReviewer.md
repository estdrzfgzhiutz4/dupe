# Media Duplicate Reviewer — Project Handoff

**Status:** Native macOS prototype in active testing.  
**Current source snapshot:** `MediaDuplicateReviewerV4/` from `MediaDuplicateReviewer_native_macOS_v4.zip`.  
**Important:** Version 4 was generated to address issues found while testing version 3, but it has **not yet been verified by the user on macOS**. Treat it as an unverified development baseline, not as a safe production release.

---

## 1. Product goal

Build a transparent, local-only macOS app that helps review and remove duplicate or inferior copies of personal photos and videos across one or more folder trees.

The app must:

- work locally on macOS without uploading media;
- show native previews only for visible review items;
- detect byte-identical duplicates reliably;
- find renamed/re-encoded visually matching photos and likely matching videos;
- help choose which copy to keep without unsafe automatic assumptions;
- show file locations clearly, because duplicates may intentionally exist in organised folders;
- move files to macOS Trash only after explicit user selection and confirmation.

The user values safety over aggressive deletion. A missed duplicate is preferable to suggesting deletion of the wrong photograph or video.

---

## 2. Security/trust requirement

The user correctly raised that “compiled locally” does **not** make downloaded source safe. A downloaded shell launcher or Swift source is executable code.

Project policy:

- Keep source visible and inspectable.
- Do not distribute opaque patching scripts that silently rewrite code on the user’s Mac.
- Prefer replacing a readable source file or making reviewable Git commits/PRs.
- Do not claim unsigned locally compiled software is inherently trusted.
- For broad distribution, eventually use ordinary macOS signing/notarization practices.
- Until the app is thoroughly tested, advise use on copied test folders and review Trash before emptying it.

## 3. Current implementation overview

The current prototype is a single-file SwiftUI application plus a zsh build launcher.

### Files expected in the repository

```text
MediaDuplicateReviewer.swift
Build and Open Media Duplicate Reviewer.command
README.md
HANDOFF.md
```

### Build output

The launcher builds and opens:

```text
~/Applications/Media Duplicate Reviewer.app
```

The launcher must compile the SwiftUI source using `-parse-as-library`, because the source contains an `@main` app entry point.

### Apple frameworks used

| Function | Framework / approach |
|---|---|
| UI | SwiftUI + AppKit |
| Native previews for visible cards | QuickLookThumbnailing / `QLThumbnailGenerator` |
| Exact byte identity | CryptoKit SHA-256 |
| Image visual confirmation | Vision / `VNGenerateImageFeaturePrintRequest` and `VNFeaturePrintObservation.computeDistance` |
| Video frame extraction | AVFoundation / `AVAssetImageGenerator` |
| Still-image metadata/dimensions | ImageIO |
| File type identification | UniformTypeIdentifiers |
| Moving files to Trash | Foundation `FileManager.trashItem` |

No HTML reports or exported thumbnail folders should be generated by the native app. Thumbnail representations are requested as visible cards appear; macOS may use its own Quick Look cache.

## 4. User workflow requirements

### Roots / scan scope

- Root A is required.
- Root B is optional.
- A single selected root must find duplicates **within that folder tree**.
- Two roots must find duplicates within either root and across both roots.
- Identical or overlapping roots should be rejected, because selecting a folder plus its subfolder can cause confusing scope and apparent duplication.
- The scan must deduplicate canonical file paths.

`Root A / Root A` results are intentional when two different files within Root A match.

### Review UI

- One native app window; no generated web pages.
- Paginate or lazily render review cards, approximately 50 visible items/pairs per page.
- Preview only visible media.
- Whole card is clickable to select the file for Trash; not only a tiny checkbox.
- Show **Open** and **Show in Finder** for originals.
- Show full meaningful relative paths, not only Root A / Root B labels.
- Keep a Selected tab/list.
- Move selected files directly to Trash after a confirmation dialog.

## 5. Matching model

### A. Exact duplicates — definitive

Exact duplicates are files with the same SHA-256 content hash.

Desired behaviour:

1. Scan the entire combined collection.
2. Group files by byte size first to avoid hashing obviously unique sizes.
3. Hash only size-collision candidates.
4. Group matching hashes across and within roots.
5. Present all locations in each exact group.

Exact groups are the only items that may be called definite duplicates.

### Exact-copy selection controls

Required controls:

- **Keep one per group, prefer Root A**
- **Keep one per group, prefer Root B** when B exists
- **Keep first sorted path per group**
- Manual per-file selection

Safety:

- Never allow every item in an exact group to be moved to Trash.
- Display paths because copies may be deliberately stored in editing/sorted folders.

### B. Similar photos — review candidates only

Non-byte-identical photos may still be converted, resized, recompressed, renamed or exported copies.

Current intended pipeline:

1. Exclude exact-duplicate items from non-identical photo matching.
2. Generate a cheap perceptual fingerprint (`dHash`) per image.
3. Index fingerprints with a Hamming-distance near-neighbour structure (current implementation uses a BK tree).
4. Add metadata-based and related-filename candidate paths, but never treat those as proof.
5. Confirm candidate pairs using Apple Vision feature-print distance.
6. Present confirmed visual pairs for review.

There is also an optional exhaustive photo scan that compares all non-identical image pairs with Vision. It is intentionally slower and should be clearly labelled.

Important design point: a “visual hash” is not a normal duplicate hash. Converted copies are generally **near** in perceptual-feature space, not byte-for-byte identical keys. Efficient near-neighbour indexing avoids comparing every file to every other by default.

### C. Similar videos — review candidates only

Exact identical videos belong under Exact Copies.

For non-identical video candidates:

1. Extract frames at approximately 10%, 50% and 90% using AVFoundation.
2. Create cheap perceptual fingerprints for those frames.
3. Use indexed near-neighbour matching to narrow candidates.
4. Confirm with Vision feature-print comparison for corresponding frames.
5. Require at least two of three sampled frames to match before listing a video pair.
6. Ask for playback inspection before deletion.

This replaces the earlier overly broad duration/aspect-only candidate generation, which produced a very large set for short Live Photo-style `.MOV` files.

## 6. Metadata use

Metadata is valuable evidence and a candidate-finding aid, but it must not prove deletion on its own.

Useful metadata:

- EXIF original capture date
- Camera model
- Pixel dimensions
- Orientation / rotated-equivalent dimensions
- Video duration
- Video dimensions
- Relative path
- File format

Current source reads image capture date and camera model via ImageIO. Video metadata is currently much more limited.

Potential later improvements:

- video creation/capture metadata;
- codec and bitrate display for non-identical videos;
- more robust orientation metadata display;
- persistent scan cache only if needed for speed/resume.

## 7. Keeper suggestion rules

A suggestion is allowed only **after** visual confirmation; it is not an automatic deletion decision.

### Intended photo rules

For an isolated strong visual match:

| Condition | Suggested keeper / Trash candidate |
|---|---|
| Same effective dimensions after allowing width/height rotation; HEIC vs JPEG | Prefer keeping HEIC; suggest JPEG for Trash, while noting compatibility considerations |
| Same format and same rotation-equivalent pixel dimensions; size differs | Prefer keeping larger file; suggest smaller file for Trash even if the difference is only a few KB, but keep the warning that metadata can affect size |
| Clearly lower resolution with compatible aspect ratio | Suggest lower-resolution file for Trash |
| PNG vs JPEG/HEIC without a clear quality basis | Review only; do not assume larger PNG is superior |
| Aspect ratio materially differs | Do not make a keeper suggestion merely from name or size |

Version 4 additionally restricts automatic photo suggestions to very strong Vision matches (`distance <= 0.060`). This threshold is a current implementation choice and should be tested, not assumed final.

### Rotation/orientation

The user found a matching photo pair where one was `1936 × 2592` and the other `2592 × 1936`. These must be treated as equal dimensions after orientation/rotation for keeper suggestions.

### HEIC

The user prefers HEIC as keeper over JPEG when the app has confirmed the same photo and effective resolution. Keep this as a suggestion, not a silent automatic deletion, because compatibility and conversion history may matter.

## 8. Multi-variant overlap: highest-priority safety issue

A single image can match multiple variants, for example:

```text
A matches B
A matches C
```

The earlier pair-based auto-selection logic could select A as inferior in one row and then select another alternative in another row, creating a dangerous state where both sides of a visual relation were selected.

### Version 3 observed failure

The user saw the same physical file appear in multiple rows. Selecting a file in one row selected that same item in another row, and automatic suggestions could produce apparently contradictory selections.

### Version 4 intended mitigation

- Count how many confirmed rows each file appears in.
- If either side of a pair participates in multiple visual-match rows, show the pair but **suppress automatic selection**.
- Before moving selected files to Trash, block the action if both sides of any displayed visual-match row are selected.

### Recommended architectural next step

Replace pair-only presentation for overlapping matches with a **variant cluster** model:

1. Build a graph where nodes are files and edges are confirmed visual matches.
2. Present each connected component as one group of variants.
3. Let the user choose a single keeper or multiple keepers for that group.
4. Suggest candidates only at the group level, never independently per overlapping pair.

This is safer and more intelligible than displaying repeated overlapping pair rows.

## 9. Live Photo / companion handling

The user has Live Photo-like exports: a still photo and a `.MOV` motion component.

Requirements:

- Never compare images against videos as duplicate alternatives.
- Show a warning when moving only one component of a likely still/video companion pair.
- No permanent database is required for initial implementation.

### Current implemented heuristic

Version 4 maps an image and a video as possible companions when they share:

- the same folder; and
- the same basename, differing only by extension.

Example:

```text
IMG_6715.HEIC
IMG_6715.MOV
```

This is deliberately labelled **Possible Live Photo companion** and is only an in-memory warning.

### Future enhancement

If feasible, read and correlate genuine Live Photo content identifiers rather than relying only on same-folder/same-basename heuristics. Do not claim this is already implemented.

## 10. Real user test cases and expected outcomes

These examples came from actual testing and should be preserved as regression scenarios.

### A. Same photo, same JPEG resolution, materially different size

```text
CIMG0847_Original.jpg   1,299,611 bytes   3264 × 2448
CIMG0847_Original.JPG     923,193 bytes   3264 × 2448
```

Expected:

- Must first be visually confirmed.
- Then suggest keeping the larger JPEG / trashing the smaller JPEG.

### B. Same stem but different actual images

```text
IMG_0160.JPG   3264 × 2448
IMG_0160.PNG   1242 × 2688
```

Observed in an early report version: incorrectly suggested lower-resolution deletion solely from name association.

Expected:

- Filename alone must never make this a deletion candidate.
- If Vision does not confirm the same content, it should not appear as a confirmed similar-photo pair.

### C. Strong confirmed same JPEG; suggestion was missing

```text
IMG_4431_Original.jpg   933 KB   3888 × 2592
IMG_4431_Original.JPG   659 KB   3888 × 2592
Vision distance: 0.0168
```

Expected:

- Suggest trashing the smaller JPEG.

### D. Same photo with swapped dimensions/orientation

```text
185_Original.jpg   1.4 MB   1936 × 2592
IMG_0721.JPG       1.8 MB   2592 × 1936
Capture date: 2011:01:16 00:53:34
Camera: iPhone
Vision distance: 0.0409
```

Expected:

- Treat dimensions as equivalent after rotation.
- Suggest keeping the larger JPEG, subject to visual review.

### E. Same-root match with tiny size difference

```text
Root A / Root A
IMG_2112_Original.jpg     375 KB   1600 × 1200
iphone 150_Original.jpg   377 KB   1600 × 1200
Capture date: 2010:02:28 10:51:13
Camera: iPhone 3G
Vision distance: 0.0000
```

Expected:

- Same-root matching is intentional.
- For a visually identical same-format/effective-resolution pair, suggest keeping the slightly larger file even though the difference is under 3%.

### F. Overlapping variants / unsafe pair selection

```text
IMG_0239_Original 2.jpg   248 KB   1280 × 960
IMG_0239_Original.jpg     251 KB   1280 × 960
IMG_0239.JPG              387 KB   1280 × 960
```

Observed:

- One physical file appeared in multiple rows.
- Pair-based suggestions risked selecting conflicting sides.

Expected:

- No automatic suggestion in overlapping visual-match rows until a group-level keeper is chosen.
- Trash action must refuse if both sides of any confirmed relation are selected.
- Preferred future UI: show all three in one variant group.

### G. Video candidate explosion

Observed while analyzing non-identical videos:

```text
Video comparisons: 61 of 19174...
```

Likely cause:

- Many short videos/Live Photo motion clips shared matching duration/aspect candidate criteria.

Expected after candidate-index revision:

- Use fingerprints from 10%, 50%, 90% frames to narrow candidates before Vision confirmation.
- Dramatically reduce unnecessary pair confirmations.

### H. Possible Live Photo companion

```text
IMG_6715.MOV
Possible Live Photo companion: IMG_6715.HEIC
```

Expected:

- Continue to warn when only one component is selected for Trash.

## 11. Current Version 4 implementation status

Version 4 was generated to incorporate:

- orientation-equivalent dimensions;
- strong same-format same-dimension suggestions even below 3% size difference;
- HEIC-over-JPEG preference for confirmed same-image/effective-resolution pairs;
- suppression of suggestions on overlapping visual pairs;
- final Trash-time protection against both sides of a visual pair being selected;
- indexed three-frame video candidates rather than duration-only candidate explosion;
- exact duplicate scanning within and across selected roots;
- possible Live Photo companion warnings.

### Not yet verified

At the time of this handoff:

- Version 4 has **not** been confirmed to compile/run successfully on the user’s Mac.
- Its fixes have **not** been verified against the regression cases above.
- The cluster-based redesign for overlapping similar variants has **not** been implemented; version 4 only suppresses suggestions and blocks dangerous final deletion states.

Do not treat v4 as deletion-safe until it has been built and tested using copied folders.
